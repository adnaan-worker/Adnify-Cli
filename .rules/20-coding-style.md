# 20 Coding Style

## 1. 技术栈约束

- 包管理工具统一使用 `bun`
- 运行时以 `Bun + TypeScript` 为主
- CLI 表现层统一使用 `Ink`
- 不随意混入与当前架构不匹配的重型框架

## 2. 命名规范

### 文件命名

- 领域模型文件使用语义名
- 聚合根、实体、服务、用例使用 `PascalCase.ts`
- Hook 使用 `useXxx.ts`
- 配置和常量文件使用清晰名词，不使用模糊缩写

### 类型命名

- 接口、类型、类名必须体现业务语义
- `Props`、`Snapshot`、`Command`、`Result` 这类后缀可以使用
- 禁止出现 `Data`, `Info`, `Temp`, `UtilThing` 这类弱语义命名

### 变量命名

- 变量名必须能表达用途
- 禁止大量使用 `data`, `item`, `obj`, `tmp`, `res2`, `finalData`
- 循环变量可以短，但不能失去语义

## 3. 注释规范

本项目要求中文注释，但注释必须有信息量。

### 应该写注释的地方

- 聚合根职责
- 用例职责
- 适配器职责
- 复杂条件分支
- 未来替换点

### 不应该写的注释

- “给变量赋值”
- “循环数组”
- “调用函数”
- 明显从代码表面就能看出的废话

### 注释要求

- 简短
- 准确
- 解释意图，不重复代码表面含义

## 4. 函数规范

- 单个函数优先控制在 40 行以内
- 一个函数只做一件主事
- 如果一个函数需要大量注释才能看懂，优先拆分
- 参数超过 4 个时，优先考虑对象参数

## 5. 组件规范

### Ink 组件

- 组件只负责渲染和轻量交互
- 复杂逻辑下沉到 Hook 或应用层
- 不把大段业务流程直接写进组件 JSX 周围

### Hook

- Hook 负责表现层状态桥接
- Hook 不直接承担基础设施细节
- Hook 不直接承载领域规则

## 6. 错误处理

- 不允许静默吞错
- 用户可见错误要给出明确状态文本
- 基础设施错误要带上下文
- 能恢复的错误优先返回结果，不直接中断全流程

## 7. 复用与抽象

### 允许抽象的前提

- 至少出现第二个真实使用点
- 抽象后语义更清晰
- 抽象后不会增加理解成本

### 禁止的抽象

- 只为“看起来高级”而创建中间层
- 只服务单个调用点却被包装成通用框架
- 提前为未来假想需求搭很多空接口

## 8. 禁止事项

- 不复制粘贴同一逻辑到多个文件
- 不随意新增 `index.ts` 做无意义转发
- 不在一个文件里同时定义 UI、业务模型、仓储实现
- 不引入未验证收益的重依赖
- 不保留未使用导入、未使用类型、未使用函数
