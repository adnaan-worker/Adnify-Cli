# 00 Core

## 1. 项目定位

`Adnify-Cli` 是一个基于 `Bun + TypeScript + Ink` 的 CLI AI 编程助手项目，采用 `DDD` 作为核心组织方式。

这个项目允许高效试错，但不允许失控开发。

“vibecoding” 在本项目中的定义是：

- 允许快速原型
- 允许大胆尝试
- 允许边做边演进
- 但必须遵守边界、命名、验证和清理规则

## 2. 总目标

所有开发都必须服务于以下目标：

1. 架构清晰，不把临时方案写成长期负债。
2. 高性能优先，不引入无意义重量依赖。
3. 低耦合高内聚，不让一个模块同时承担多种职责。
4. 高复用低冗余，不重复造同类模型、工具、适配器和流程。
5. 对 AI 友好，但不能被 AI 生成内容反向拖垮代码质量。

## 3. 总规则

### 必须遵守

- 优先做最小闭环，再做扩展能力。
- 新增代码必须能说明它属于哪个层、哪个上下文、为什么存在。
- 新功能必须优先复用现有模型、端口和适配器。
- 写完后必须检查是否产生了重复类型、重复 helper、重复流程。
- 文档、目录和代码必须一致。

### 明确禁止

- 为了图快把业务规则直接写进 `presentation`。
- 为了图快把 Node 文件系统、终端 IO、网络调用直接写进 `domain`。
- 建立没有使用价值的“万能 utils”。
- 创建“BaseService` / `CommonService` / `HelperService`”这类模糊抽象。
- 同一语义创建多个近似类型，只因当前文件引入不方便。
- 不经清理地保留注释掉的大段废弃代码。

## 4. Vibecoding 约束

### 可以快，但不能乱

- 可以先做 stub，再替换成正式实现。
- 可以先做内存仓储，再替换成持久化仓储。
- 可以先做同步响应，再升级成流式响应。

### 但必须满足

- stub 必须放在正确层级。
- 未来替换点必须明确。
- 不允许把“临时实现”写成“全项目依赖的硬耦合实现”。

## 5. 决策顺序

遇到方案冲突时，按以下顺序决策：

1. 先保证领域边界正确。
2. 再保证可维护性。
3. 再保证性能。
4. 最后才是短期开发便利。
